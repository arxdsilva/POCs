3. Why do we have a scheduler? --------------------------------

Everyone here knows that 
Golang uses goroutines
to manage the workload 
of a program and to do concurrency.

Threads have many benefits, 
such as their own signal mask,
can be assigned to CPU affinity, 
put to cgroups and queried
for the resources that they use.

All this are just a few of its
features,

And they are very good but 
at the same time
they are too many overheads
that Go programs do not need.

Another issue is that the OS 
cannot make informed scheduling decisions. 

For example, the Go garbage collector
requires all threads to be stopped 
and that the memory is at a consistent state.

This involves waiting 
for running threads 
to reach a point where 
we know that memory is consistent.

When you involve many 
random threads in this equation,
it is highly possible that 
you will need to wait 
a very long time 
before being able to stop it.

> --9-- < slide change

With the Go scheduler, 
it can only schedule
the garbage collection on points that it knows 
that the memory is consistent. 

This means that 
when stopping for a garbage collection,
we only wait for the threads that are being
actively ran on a CPU core.

> --10-- < slide change

It is also important to note that goroutines
are lighter on the memory 
and faster to create, 
destroy and switch context 
than usual OS threads.

4. When does it schedule ? --------------------------------

So, when does it schedule?

Conceptually,
It could be any time 
that a program should or would affect 
a goroutine execution.

Could be any of, but not limited to:
1. goroutine creation;
2. goroutine blocking;
3. blocking system call;
4. network i/o;

note that this is a active
work from the goroutine.


> --14-- < slide change

In order to do all that we discussed
it is vital that the scheduler
has clear priorities. 

The first is to use 
as little as possible kernel threads,
as we've seen that 
they are expensive to create 
and manage.

Secondly, as Go is intended for 
highly performant programs,
so we want to support high concurrency. 
    This means that Go programs 
    should be able to run
    lots of goroutines.

Lastly, it is essential 
that we can scale to any number of cores,
    so that parallelism can be achieved
    in any amount of goroutines.


> --15-- < slide change

But how does it work, right?

So recapitulating, whenever operations 
should or would affect 
a goroutine execution occurence,

the runtime causes a 
switch to create a new 
goroutine in a thread.

As seen in this image 
    - used from that image processor -

in the main thread,
g_main is executed and keeps running.

In the case of a channel receive
however we want the goroutine to block.

So the context is switched out
for g1.

So now that we got through why, when and what,
lets start building with that example.

The next step would be on 
How do we multiplex goroutines
onto kernel threads.

well, multiplexing is a really hard task 
and the best way to answer

into two different problems:

First, when to create threads?

Secondly, how do we distribute 
goroutines across threads?

In order to start answering
these questions, 
I must present you 
one important structure first.

> --17-- < slide change

The runqueue 
is a heap allocated 
First-In-First-Out queue
that stores the goroutines 
informations for our processors.

it works the same as any 
FIFO queue you can imagine.

When there is a create event, 
a new goroutine is added 
to the end of the queue.

And when it is time 
to run a goroutine, 
you run it through the head 
of the queue.

2. create and destroy threads per goroutine

3. Now about the goroutine distribution:

 threads.

With this "handoff" mechanism,

This all works as long as

What happens when we have a goroutine
that does not cooperate,
and does not do any blocking?

Lets talk about this
complicatedAlgo function.

If it is long running, CPU bound

what happens?

If the scheduler isnt ever

It will lead to

> --59-- < slide change

In order to fix that, 
the scheduler implements
asynchronous preemption.

Since Go 1.14,
this preemption is signal-based,
which is more sophisticated
than simple polling.

Essentially:

A background thread
called "sysmon" monitors
long-running goroutines.

When a goroutine has been running
for longer than 10 milliseconds,
sysmon sends a signal (SIGURG on Unix)
to unschedule it.

Once unscheduled,
It would be unwise 
to just put it back 
on any other runqueue.

So these special goroutines
go to a global runqueue.

> YES, strange right?

This exists in addition 
to the distributed runqueues.

To avoid contention,
this is a lower priority queue,
so threads will access it
a lot less offently.

With that we have the main 
ideas and tricky situations
that the scheduler implements.

These decisions also leads us
to a number os limitations.

1. It has No clear priorities:
    as it uses a First-In-First-out queue
2. Also It has No strong preemption:
    there is no fairness or latency
    guarantees.

    So it is entirely possible

    to bring the entire system
    to slow down to a halt.
3. scheduler doesnt know the system hardware:
    no real localization 
    between the data and 
    the goroutine computation

    for example, in the new intel CPUs
    that has efficiency and 
    performance cores, it will not
    differentiate them when scheduling
